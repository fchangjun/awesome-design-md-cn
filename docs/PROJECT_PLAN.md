# awesome-design-md-cn 改造方案

## 1. 项目目标

将 `awesome-design-md` 改造成一个更适合中文用户使用的设计系统导航项目。

最终形态采用“双形态”：

- 作为 GitHub 仓库，承载 `DESIGN.md` 内容、预览页和元数据
- 作为静态站点，提供中文首页、搜索、分类、预览和使用指引

这个项目的重点不是在线生成 UI，而是帮助用户更快找到合适的设计风格，并把这些设计风格交给 AI Agent 使用。

## 2. 改造边界

### 做什么

- 新建中文项目仓库 `awesome-design-md-cn`
- 保留原始设计资产的目录组织方式
- 新增中文说明、中文标签、中文搜索入口和中文使用指引
- 为站点增加适合中文用户的筛选和展示方式
- 后续新增中国语境下常见的设计风格参考

### 不做什么

- 不批量重写原始 `DESIGN.md`
- 不覆盖原项目已有品牌目录和原始预览文件
- 不把项目直接改造成重前端应用或在线生成器
- 不在第一阶段引入后端服务

## 3. 用户问题定义

当前原项目对中文用户的主要问题：

- 入口说明偏英文语境，用户不容易快速理解 `DESIGN.md` 的用途
- 导航方式偏品牌名检索，不适合中文用户按场景找风格
- 缺少中文语境下的使用示例、提示词和标签
- 缺少中国用户熟悉的平台、品牌和业务场景映射

## 4. 产品形态

推荐形态：静态展示站 + GitHub 内容仓库

原因：

- 内容型项目，核心诉求是浏览、筛选、理解和跳转
- 静态页面成本低、可直接部署、维护简单
- 可保留仓库作为内容源，降低后续扩展成本
- 本地打开即可预览，也适合 GitHub Pages / Vercel 部署

## 5. 信息架构

第一版建议包含以下内容：

- 首页
- 设计系统列表页
- 单个设计系统说明页
- 原始预览入口
- 使用说明页

首页重点模块：

- 项目介绍
- 新手快速上手
- 热门风格推荐
- 分类筛选
- 搜索框
- 最近新增或推荐内容

设计系统卡片建议字段：

- 英文名
- 中文名
- 风格简介
- 适用场景
- 中文标签
- 行业分类
- 明暗偏好
- 推荐提示词
- 预览入口

## 6. 内容策略

### 6.1 原始资产策略

保留原有目录结构，例如：

- `design-md/notion`
- `design-md/stripe`
- `design-md/linear.app`

不直接改写其中核心资产，尽量通过新增元数据和外层页面来增强体验。

### 6.2 中文元数据策略

新增统一元数据文件，建议命名为：

- `data/designs.json`

建议字段：

- `slug`
- `name`
- `nameZh`
- `summaryZh`
- `category`
- `tagsZh`
- `styleKeywords`
- `aliases`
- `useCases`
- `recommendedPromptZh`
- `isDarkFriendly`
- `previewLight`
- `previewDark`

这样可以避免把中文文案写死在页面脚本中，后续扩展也更稳定。

### 6.3 中文使用指引

需要补齐以下中文内容：

- `DESIGN.md` 是什么
- 适合谁用
- 怎么给 Cursor / Claude / Codex 使用
- 如何选择适合自己的风格
- 常见问题

## 7. 首页改造方向

当前首页是简单的静态卡片列表。改造后建议具备：

- 中文优先标题和说明
- 中英文名称同时展示
- 中文搜索和别名搜索
- 按行业、风格、工具类型筛选
- 按“适合做什么”筛选
- “复制中文提示词”入口
- 推荐入口，例如“适合后台系统”“适合官网”“适合 AI 产品”

## 8. 中国语境扩展

第二阶段逐步补充中国用户更熟悉的风格参考。建议优先顺序：

- 飞书
- 微信 / 企业微信
- 支付宝
- 钉钉
- 小红书
- 抖音
- B 站
- 美团
- 淘宝 / 天猫
- 京东

说明上需要明确：

- 这些内容是基于公开页面提炼的风格参考
- 不代表官方设计系统

## 9. 实施阶段

### Phase 1: 初始化与信息重构

- 建立新仓库
- 输出项目 README
- 输出改造方案文档
- 定义目录结构和元数据方案

### Phase 2: 静态首页中文版

- 建立新的中文首页
- 抽离首页中的硬编码数据
- 增加中文搜索、分类、标签和筛选
- 优化首页视觉层级和信息密度

### Phase 3: 内容接入

- 接入原始 `design-md` 内容
- 为每个设计系统补充中文元数据
- 增加推荐提示词和适用场景

### Phase 4: 扩展与部署

- 增加更多中国语境风格
- 适配 GitHub Pages / Vercel 部署
- 根据使用反馈再决定是否框架化

## 10. 建议目录结构

```text
awesome-design-md-cn/
├── README.md
├── docs/
│   └── PROJECT_PLAN.md
├── data/
│   └── designs.json
├── design-md/
├── index.html
├── assets/
└── scripts/
```

第一阶段只需要先落 `README.md` 和方案文档，后续再逐步补齐 `data/`、`design-md/` 和首页实现。

## 11. 实施原则

- 先保证信息架构正确，再做视觉细节
- 先做静态页面和内容组织，不急着上框架
- 先解决中文用户理解和检索问题
- 所有改造以新增为主，避免破坏原始内容

## 12. 下一步执行建议

下一步直接进入首页和数据层初始化，优先做：

- 建立 `data/designs.json`
- 重写 `index.html` 为中文版导航首页
- 设计卡片展示字段和筛选逻辑
- 规划如何从原项目迁移 `design-md` 内容

## 13. 后续视觉与交互待办

以下事项属于首页后续迭代 backlog，当前先记录，不在这一轮继续实现。

### 13.1 首页信息架构

- 将首页控制在“推荐起点 + 设计系统库”两层核心内容，避免无限下拉的内容堆叠
- 评估首页默认只展示部分结果，例如前 `12` 个，并补“查看更多”或分页
- 减少首页承担的信息量，把完整说明、摘要和补充信息更多放到详情页

### 13.2 设计系统库展示方式

- 评估将“设计系统库”从卡片流改为更紧凑的列表型布局
- 如果继续保留卡片，进一步弱化卡片感，让库区更像连续目录而不是一组独立白盒
- 根据桌面端宽度重新评估三列 / 四列的密度平衡，而不是固定使用单一列数

### 13.3 卡片信息密度

- 继续压缩普通库卡的信息层级，只保留最关键的判断信息
- 优先保留：中文名、英文名、分类、一个代表性场景、两个操作入口
- 谨慎保留摘要和标签，避免每张卡片重复输出过多文字
- 精选卡和普通库卡继续区分视觉语言，不让两者看起来像同一层级

### 13.4 筛选器体验

- 继续优化左侧筛选栏，让其更像正式产品中的筛选器，而不是设置面板
- 增加更明确的筛选状态反馈，例如当前条件、已筛结果数、清空后状态恢复
- 评估是否加入排序、只看精选、按分类折叠等能力，降低浏览压力
- 保持桌面端 `sticky` 侧栏方案，移动端继续采用顶部单列结构

### 13.5 视觉舒适度

- 继续降低大面积纯白造成的视觉疲劳，保持浅冷灰底和更柔和的面板白值差
- 只保留克制的冷蓝强调色，避免重新引入高饱和彩色块
- 继续收卡片边框、hover 和操作区的重量，让整页更安静、更耐看
- 所有视觉迭代优先以“减噪”为原则，而不是增加新装饰
